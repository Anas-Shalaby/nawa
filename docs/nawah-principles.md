# Nawah Decision Framework

> Version 1.0
> Last Updated: July 2026

---

# Why This Document Exists

Every product becomes more complicated over time.

Every week brings new feature requests.

Every customer wants something different.

Without clear decision-making principles...

the product slowly becomes cluttered.

This document protects Nawah from becoming just another clinic management system.

Whenever someone proposes a new feature...

this document decides whether it belongs in Nawah.

---

# Our Mission

Nawah exists to help clinics operate with less stress.

Everything we build must reduce chaos.

Not create more of it.

---

# Our Product Vision

Nawah is not software.

Nawah is the operating system of a clinic.

The product should disappear into the background.

Doctors should think about patients.

Receptionists should think about people.

Owners should think about growth.

Nobody should think about the software.

---

# The Five Filters

Every new feature must pass all five filters.

---

## Filter 1

Does it solve a real daily problem?

Bad

"We can build this."

Good

"Clinics struggle with this every day."

---

## Filter 2

Does it reduce work?

Every feature must remove work.

Not create work.

If users need additional training...

we probably designed it wrong.

---

## Filter 3

Does it reduce decisions?

Software should make decisions easier.

Never harder.

Users should spend their attention on patients.

Not on menus.

---

## Filter 4

Does it simplify the workflow?

If a feature introduces

- extra screens
- extra forms
- extra clicks

without significant value...

don't build it.

---

## Filter 5

Would 80% of clinics benefit?

Nawah is a platform.

Not a custom software project.

We optimize for common workflows.

Not edge cases.

---

# The 80/20 Rule

Always optimize for the majority.

Never redesign the entire product for one customer.

Edge cases should be optional.

Core workflows should stay simple.

---

# Feature Categories

Every feature belongs to exactly one category.

## Core

Without it...

Nawah cannot operate.

Examples

- Booking
- Queue
- Patients
- Visits

---

## Growth

Makes clinics more successful.

Examples

- Reports
- Analytics
- Team
- Inventory

---

## Delight

Improves experience.

Examples

- Animations
- Keyboard shortcuts
- Better search

---

## Future

Interesting ideas.

Not priorities.

Examples

- AI
- Marketing Automation
- Insurance Integrations

---

# What We Never Build

We don't build features because competitors have them.

We don't build features because they look impressive.

We don't build features that create unnecessary complexity.

We don't build settings that only one customer understands.

---

# The Simplicity Rule

If a feature requires:

- a long explanation
- multiple tutorials
- onboarding videos

Then the design is probably wrong.

---

# Navigation Rule

A user should always know:

Where am I?

What should I do?

What happens next?

If not...

the workflow is broken.

---

# Permission Rule

Permissions exist to simplify work.

Not only to secure it.

Every role should see less.

Not more.

---

# Dashboard Rule

Dashboards are not decoration.

Every widget must answer one question.

Every KPI must support one decision.

If a widget does nothing...

remove it.

---

# Forms Rule

Forms should ask only for information needed now.

Advanced settings stay hidden until necessary.

---

# Data Rule

Never ask users to enter data twice.

Reuse existing information whenever possible.

---

# Automation Rule

If users repeat the same action many times...

consider automation.

If automation creates confusion...

don't automate.

---

# Visual Rule

Whitespace is a feature.

Typography is a feature.

Hierarchy is a feature.

Beauty comes from clarity.

Not decoration.

---

# Performance Rule

Speed is part of the product.

Fast software feels simpler.

Slow software feels complicated.

---

# Error Rule

Errors should explain

- what happened
- why
- how to fix it

Never blame users.

---

# Product Review Checklist

Before approving any feature ask:

□ What problem does this solve?

□ Who needs it?

□ Which workflow changes?

□ Which role benefits?

□ How many clicks are removed?

□ How much thinking is removed?

□ Is this useful every week?

□ Can we remove something else because of it?

If these questions cannot be answered...

the feature is not ready.

---

# Success Metrics

A successful feature should:

✔ Reduce clicks

✔ Reduce navigation

✔ Reduce training

✔ Reduce support questions

✔ Reduce stress

✔ Increase speed

✔ Increase confidence

---

# Our North Star

Every release should make the clinic feel calmer than before.

Not more powerful.

Calmer.

If a feature makes the software feel heavier...

it doesn't belong in Nawah.
